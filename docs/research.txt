GC choice
My recommendation: mark-sweep first
Why
For a small board, copying GC doubles effective heap requirement. The Cheney variant is elegant, but it requires two semispaces. 

A single-heap mark-sweep design is friendlier if RAM is tight. The non-Cheney small Lisp version also emphasizes a single cell[] memory array. 

Tradeoffs
Mark-sweep

pros: single heap, stable pointers, simpler foreign references

cons: allocation slower, fragmentation concerns, mark phase complexity

Copying/Cheney

pros: very simple allocation, compaction “for free”

cons: needs 2x semispace, moving objects complicate root handling

For your board demo, I would pick:

mark-sweep now

maybe compact strings separately later

maybe copying collector only in emulator or larger-memory builds

1. Reset / entry point
You need one small assembly file that:

sets the initial stack pointer

clears .bss if your runtime does not already do it

optionally copies .data from ROM to RAM

jumps to main()

This is the most important assembly file.

2. UART / console primitives
If there is no usable C runtime or no libc, expose a tiny board/emulator I/O surface:

putchar

optional getchar

optional puts or write

If you already have these in C and they work reliably, do them in C instead.

3. Trap / halt / panic
A tiny assembly or mixed C/asm routine for:

halt

breakpoint

panic loop

Very useful during bring-up.

Optional assembler, later
Only after the interpreter works:

tagged-value tests

cons allocation fast path

primitive arithmetic helpers

GC scan helpers

context save/restore if you later do trampolines or coroutines

Do not start with these.
